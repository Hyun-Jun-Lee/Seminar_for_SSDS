배경: 왜 “언어쌍”이 도메인이 되어야 했나

기존 Curitalk는 구조상으로는 꽤 일반적인 대화 학습 앱처럼 보였습니다. 사용자는 로그인하고, 관심 주제를 검색해 Topic Prep 카드를 만들고, Free Chat이나 Roleplay로 대화를 시작합니다. 대화가 이어지는 동안 메시지는 저장되고, 문법 피드백은 비동기 작업으로 생성됩니다.

하지만 중요한 가정 하나가 코드 곳곳에 박혀 있었습니다. 프롬프트는 “English conversation assistant”라고 말했고, 문법 피드백은 영어의 주어-동사 일치와 질문 어순을 중심으로 검사했으며, 온보딩과 로그인 화면도 영어회화 앱으로 자신을 소개했습니다. 데이터 모델은 언어 중립처럼 보였지만 실제 행동은 영어 전용이었던 셈입니다.

핵심 문제 언어 설정이 단순 UI 텍스트라면, 사용자가 언어쌍을 바꾼 순간 기존 대화의 성격도 바뀔 수 있습니다. 반대로 언어 설정이 대화 생성 시점에 snapshot으로 저장되면, 오래된 영어 연습 대화와 새 한국어 연습 대화가 같은 계정 안에서 안전하게 공존합니다.

초보자를 위한 큰 그림

대화형 학습 앱에는 적어도 네 종류의 상태가 섞입니다.

프로필 기본값

새 대화를 만들 때 어떤 언어쌍을 기본으로 쓸지 결정합니다.

대화 snapshot

대화가 시작된 순간의 언어쌍입니다. 이후 프로필이 바뀌어도 유지됩니다.

프롬프트 컨텍스트

LLM에게 목표 언어, 피드백 언어, 모국어를 알려주는 실행-time 정보입니다.

이 PR은 이 세 상태를 같은 이름과 같은 구조로 연결합니다. 이름은 LearningLanguageContext입니다. 필드는 단순합니다. native_language, target_language, feedback_language. 단순하지만, 이 세 필드가 흐르는 경로는 앱 전체를 통과합니다.

Onboarding 예: zh → ko 선택
feedback: zh
Profile API PUT /auth/me/language-preferences
Profile Defaults profiles.native_language
profiles.target_language
Conversation Snapshot conversations에 시작 시점 언어 저장
Prompts 대화, 문법, 검색, Topic Prep이 snapshot 사용

이 변경 바로 전의 좁은 배경

인증은 Supabase access token으로 FastAPI 보호 API를 호출하는 구조입니다. GET /api/auth/me는 Supabase token을 검증하고 profiles row를 만들거나 갱신합니다. 대화는 ConversationService에서 생성되고, 메시지 저장 후 LLM 응답을 만들며, 문법 피드백은 백그라운드 태스크로 생성됩니다.

모바일 앱은 Splash → Onboarding → Login → Home 순서로 이동합니다. 중요한 제약은 온보딩이 로그인보다 먼저 온다는 점입니다. 즉 사용자가 언어쌍을 고르는 순간에는 아직 서버에 저장할 사용자 ID가 없습니다. 이 PR은 그 간극을 “pending local language context”로 메웁니다.

직관: 지도는 프로필에, 여권 도장은 대화에

이 리팩터의 가장 좋은 비유는 여행입니다. 프로필의 언어쌍은 앞으로 어디로 가고 싶은지 적어 둔 지도이고, conversation snapshot은 특정 여행을 시작할 때 찍힌 여권 도장입니다.

사용자가 오늘은 영어를 연습하다가 내일 한국어를 연습하고 싶어질 수 있습니다. 지도는 바뀌어도 됩니다. 그러나 어제 시작한 영어 대화의 여권 도장이 갑자기 한국어로 바뀌면 곤란합니다. 그 대화의 맥락, 문법 피드백, 검색 준비 카드, 음성 turn은 모두 어제의 언어쌍을 기준으로 해석되어야 합니다.

언어 컨텍스트 예시

{
  "native_language": "zh",
  "target_language": "ko",
  "feedback_language": "zh"
}

이 값이 프로필에 저장되면 새 대화는 한국어 연습 대화로 시작합니다. 그러면 LLM에게 가는 시스템 프롬프트는 “한국어로 대화하라”고 말하고, 문법 피드백은 조사, 어미, 띄어쓰기, 높임말 같은 한국어 문제를 검사합니다. 설명은 중국어로 받을 수 있습니다. 여기서 흥미로운 점은 모델 공급자는 아직 바뀌지 않았다는 것입니다. 모델을 바꾸지 않고도 “무엇을 해야 하는지”를 정확히 전달하는 기반을 먼저 만든 것입니다.

코드 산책: 바뀐 부분을 흐름대로 읽기

diff는 50개가 넘는 파일을 건드리지만, 구조는 비교적 또렷합니다. 먼저 공통 계약을 만들고, 저장 모델에 붙이고, API로 노출하고, 새 대화 생성 시 snapshot을 남긴 뒤, 프롬프트와 모바일 UX가 그 값을 사용하게 했습니다.

공통 계약 backend/shared/language.py와 Flutter features/language가 같은 개념을 정의합니다.
저장소 profiles에는 기본값, conversations에는 시작 시점 snapshot을 저장합니다.
API /api/auth/me/language-preferences를 추가하고 UserProfile.language를 반환합니다.
실행 흐름 conversation, grammar, search, topic prep prompt가 언어 컨텍스트를 받아 target/feedback 언어를 반영합니다.
모바일 온보딩에서 언어쌍을 고르고, 로그인 후 서버에 sync하고, Home/Profile에서 보고 바꿀 수 있습니다.

1. 백엔드: 지원 언어쌍을 닫힌 집합으로 정의

가장 먼저 생긴 것은 LearningLanguageContext입니다. 여기서 중요한 결정은 “언어 코드를 문자열로 대충 통과시키지 않는다”는 점입니다. SUPPORTED_LANGUAGE_PAIRS가 MVP에서 허용되는 네 가지 쌍을 닫힌 집합으로 갖고, Pydantic validator가 나머지를 거부합니다. 

SUPPORTED_LANGUAGE_PAIRS = frozenset(
    {
        (LanguageCode.KOREAN, LanguageCode.ENGLISH),
        (LanguageCode.ENGLISH, LanguageCode.KOREAN),
        (LanguageCode.CHINESE, LanguageCode.ENGLISH),
        (LanguageCode.CHINESE, LanguageCode.KOREAN),
    }
)

이 작은 집합은 나중에 큰 안전장치가 됩니다. 예를 들어 UI가 실수로 en → zh를 보내도, 백엔드 prompt builder까지 흘러가지 않고 API schema 단계에서 막힙니다.

2. 저장 모델: profile default와 conversation snapshot 분리

ProfileModel과 ConversationModel 모두 세 language column을 가집니다. 겉보기에는 중복처럼 보일 수 있지만, 의미가 다릅니다. 프로필은 앞으로 만들 대화의 기본값이고, conversation은 이미 시작된 대화의 계약입니다. 

class ProfileModel(Base):
    native_language = Column(String(8), default="ko", nullable=False)
    target_language = Column(String(8), default="en", nullable=False)
    feedback_language = Column(String(8), default="ko", nullable=False)

    @property
    def language(self) -> LearningLanguageContext:
        return language_context_from_values(...)

새 Alembic migration은 두 테이블에 column을 추가합니다. 또한 이전 profile migration은 SQLite fresh DB에서 foreign key constraint 변경이 실패할 수 있었기 때문에, batch mode로 수정했습니다. 개발 기본 DB가 SQLite인 프로젝트에서는 이런 부분이 꽤 현실적인 품질 문제입니다.

3. Auth API: 언어 선호는 profile boundary에서만 쓴다

새 endpoint는 좁습니다. 현재 사용자 자신의 언어 선호만 읽고 바꿉니다. email, 이름, active 상태 같은 다른 profile 필드를 같이 업데이트하지 않습니다. 

@router.get("/me/language-preferences")
def get_language_preferences(...):
    preferences = service.get_language_preferences(current_user.id)
    return SuccessResponse(data=preferences)

@router.put("/me/language-preferences")
def update_language_preferences(request, ...):
    preferences = service.update_language_preferences(current_user.id, request)
    return SuccessResponse(data=preferences)

이 구조 덕분에 언어 설정이 일급 계약이 되지만, 별도 “language domain”이 auth table을 직접 수정하는 식의 도메인 결합은 피합니다.

4. Conversation: 생성 시점에 snapshot 저장

Free Chat과 Roleplay 시작 시 current_user.language가 service로 전달되고, conversation row에 세 값이 저장됩니다. 이어 말하기는 프로필을 다시 보지 않고 conversation.language를 읽습니다. 

conversation = ConversationModel(
    ...,
    native_language=language_context.native_language.value,
    target_language=language_context.target_language.value,
    feedback_language=language_context.feedback_language.value,
)

system_prompt = self.build_system_prompt(
    ...,
    language_context=language_context,
)

이것이 이번 PR의 중심축입니다. 사용자가 profile을 ko → en에서 en → ko로 바꿔도, 이전 conversation은 자신이 태어난 언어쌍을 계속 들고 있습니다.

5. Grammar: 영어 문법 검사에서 목표 언어 검사로

문법 피드백은 가장 조심스러운 부분입니다. 영어 문법 규칙을 한국어에 적용하면 이상한 피드백이 나옵니다. 그래서 target이 한국어일 때는 별도 prompt branch가 생겼습니다. 

if language_context.target_language.value == "ko":
    return f"""Analyze the following Korean response for grammar and naturalness issues.
...
2. Check Korean-specific issues:
   - particles such as 은/는, 이/가, 을/를, 에/에서
   - verb endings and tense/aspect markers
   - spacing and spelling
   - honorific and formality fit
...
4. Explain issues in {feedback_name}."""

즉 “교정 문장은 목표 언어로”, “설명은 feedback 언어로”라는 분리가 생겼습니다. 이것은 중국어 화자가 한국어를 배울 때 특히 중요합니다.

6. Search와 Topic Prep: 검색 카드도 언어쌍을 알아야 한다

Topic Prep은 단순 번역 문제가 아닙니다. 검색 region, 검색 실패 안내, 예시 topic, 요약 언어, 첫 질문 언어가 모두 학습 경험을 좌우합니다. 이 PR은 search pipeline 전반에 language context를 통과시킵니다. 

return TopicPrepResult(
    ready=True,
    language=language_context,
    card=card,
    quality=card.quality,
)

검색 region은 아직 거칠지만 의도는 분명합니다. 중국어 사용자는 cn-zh, 영어 목표 언어는 us-en, 한국어 목표 언어는 kr-kr 힌트를 사용합니다. 모델 라우팅은 후속 과제로 남겼지만, 나중에 benchmark를 붙일 수 있도록 언어 정보가 이미 흐르고 있습니다.

7. Flutter: 로그인 전 선택을 로그인 후 저장하기

모바일에서 가장 흥미로운 부분은 순서 문제입니다. 온보딩은 로그인 전이고, 서버 저장은 로그인 후입니다. 그래서 OnboardingStorage가 pending language context를 secure storage에 저장합니다. 로그인 또는 세션 복원에 성공하면 AuthController가 pending 값을 서버로 동기화하고 삭제합니다. 

Future<void> _syncPendingLanguageContext() async {
  final LearningLanguageContext? pending =
      await _onboardingStorage.readPendingLanguageContext();
  if (pending == null) {
    return;
  }
  await _languagePreferencesRepository.updateLanguagePreferences(pending);
  await _onboardingStorage.clearPendingLanguageContext();
}

Home header는 현재 active pair를 badge로 보여주고, Account sheet에서는 나중에 언어쌍을 바꿀 수 있습니다. 저장 후에는 auth session을 다시 복원해서 UserProfile.language가 최신값으로 hydration됩니다.

중요한 경계와 의도적으로 미룬 것들

좋은 리팩터는 무엇을 바꿨는지만큼 무엇을 바꾸지 않았는지도 선명해야 합니다. 이 PR은 언어쌍의 “길”을 만들었지만, 모든 나라의 모든 학습 경험을 한 번에 완성하려 하지는 않습니다.

기존 데이터 보존 제외

사용자 결정에 따라 생산 데이터 backfill choreography는 만들지 않았습니다. 대신 fresh/reset DB와 기본값을 검증했습니다.

모델 라우팅 후속

DeepSeek/Qwen 같은 pair-specific provider 선택은 benchmark 이후로 미뤘습니다.

전체 i18n 아님

중국어 UX는 언어쌍 선택과 회복 안내 등 핵심 표면으로 제한했습니다.

검증된 것 백엔드 focused test 41개, Flutter focused test 22개, Flutter analyze, Alembic fresh SQLite upgrade, 그리고 diff whitespace check가 통과했습니다. 특히 fresh SQLite migration 실패를 수동 리뷰 중 발견해 기존 migration을 batch mode로 보정했습니다.
