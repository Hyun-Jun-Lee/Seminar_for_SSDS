오늘의 질문

Agent 실행이 오래 걸리거나 실패해도
추적 가능한 서비스로 만들려면?

사용자 질문 “DB가 느립니다. 원인을 분석해 주세요.”
운영 질문 어디까지 실행됐고, 어느 Worker와 Node에서 실패했으며, 결과는 어디에 저장됐는가?
3일차는 Agent를 “좋은 답변을 만드는 코드”가 아니라 실행, 상태, 로그, 복구가 필요한 운영 서비스로 보는 시간입니다.

Part 1

ReAct 전환 시
운영에서 달라지는 것

ReAct 운영 관점

고정 Node에서 동적 Tool Loop로 바뀝니다

기존 Node Agent

SQL Agent
  - Top SQL 조회
  - 실행 시간 변화 확인
  - 병목 후보 정리

ReAct Agent

SQL ReAct Agent
  - 현재 문제 정리
  - 필요한 Tool 선택
  - Observation 확인
  - 다음 Tool 필요 여부 판단
  - 충분하면 분석 결과 생성
ReAct 전환의 운영 포인트는 한 Node 안에서 실행 횟수와 Tool 호출 순서가 동적으로 변한다는 점입니다.

ReAct 실행 추적

run_id만으로는 부족합니다

ReAct Agent는 Node 내부에서 여러 step을 실행하므로, 로그가 step/action/observation 단위로 내려가야 합니다.

run_id=20260722_153012_abcd
node_name=sql_agent
step_id=1
action=tool_call
tool_name=query_top_sql
observation_status=success
duration_ms=820
운영자는 “어느 Node?”뿐 아니라 “어느 step, 어떤 Tool, 어떤 Observation 이후?”를 볼 수 있어야 합니다.

ReAct 실행 예산

자율성에는 제한이 필요합니다

react_config = {
    "max_steps": 6,
    "max_tool_calls": 5,
    "tool_timeout_sec": 10,
    "total_timeout_sec": 60,
    "allowed_tools": [
        "query_metric",
        "query_top_sql",
        "query_lock",
    ],
}
max_steps추론과 Tool 호출 반복 제한
tool_timeout느린 Tool 하나가 전체 실행을 막지 않게 함
total_timeoutWorker 점유 시간 제한
allowed_tools호출 가능한 Tool 범위 제한

Tool 정책

Agent가 Tool을 고르더라도, 허용 범위는 시스템이 정합니다

Tool 유형예시운영 정책
Read ToolMetric 조회, Top SQL 조회, Lock 조회ReAct Agent가 직접 호출 가능
Diagnostic Tool실행 계획 조회, 세션 상세 조회timeout과 argument validation 필요
Action Toolsession kill, parameter 변경, restart기본적으로 직접 실행 금지
ReAct Agent의 자유도는 Tool whitelist, timeout, 입력 검증, 권한 정책 안에서만 허용해야 합니다.

로그 마스킹

내부 추론 원문을 그대로 남기지 않습니다

운영 로그에는 모델의 사고 과정을 전문으로 저장하기보다, 판단 요약과 선택한 Tool, 관측 결과 요약을 남깁니다.

reason_summary="SQL 병목 가능성이 높아 Top SQL 조회가 필요함"
action="tool_call"
tool_name="query_top_sql"
observation_summary="elapsed time 상위 SQL 3건 확인"
ReAct 운영 로그는 디버깅 가능해야 하지만, 민감 정보와 불필요한 중간 추론을 축적하지 않아야 합니다.

Celery

긴 Agent 실행은 Worker에서 처리합니다

@celery_app.task(bind=True, acks_late=True)
def execute_agent_task(self, run_id: str):
    mark_running(run_id)

    try:
        request = load_run_request(run_id)
        result = graph.invoke({
            "user_question": request.question,
            "run_id": run_id,
        })
        save_result(run_id, result)
        mark_complete(run_id)

    except Exception as exc:
        save_error(run_id, str(exc))
        mark_failed(run_id)
        raise
Worker는 실행 시작, 성공, 실패를 모두 상태와 로그로 남겨야 합니다.

Celery 설정

설정은 운영 동작을 바꿉니다

설정의미운영 영향
task_acks_late=True작업이 끝난 뒤 ackWorker 장애 시 재전달 가능
worker_prefetch_multiplier=1Worker가 미리 많이 가져가지 않음긴 작업에서 작업 쏠림 감소
task_ignore_result=TrueCelery result backend에 결과 저장 안 함결과는 별도 DB에 저장
task_time_limittask 최대 실행 시간무한 실행 방지
acks_late는 메시지를 잃지 않게 하지만, 같은 run_id가 다시 실행될 가능성도 만듭니다.

Part 3

중복 실행과
실패 대응

중복 실행

중복 실행은 버그가 아니라 운영 위험입니다

1. FastAPI가 run_id=R1 작업을 발행
2. Worker A가 R1을 가져감
3. Worker A가 LangGraph 실행 중 종료
4. RabbitMQ는 ack를 받지 못함
5. RabbitMQ가 R1 메시지를 다시 전달
6. Worker B가 R1을 다시 실행
retry 설정을 명시하지 않아도 Queue와 Worker 구조에서는 같은 작업이 다시 실행될 수 있습니다.

Idempotency

같은 run_id가 다시 들어와도 안전해야 합니다

def mark_running_if_allowed(run_id: str) -> bool:
    run = get_run(run_id)

    if run.status in ("COMPLETE", "CANCELLED"):
        return False

    if run.status == "RUNNING":
        return False

    update_status(run_id, from_status="PENDING", to_status="RUNNING")
    return True
이미 완료된 결과를 덮어쓰지 않게 하는 것이 핵심입니다.

실패 대응

실패는 나중에 찾을 수 있게 남겨야 합니다

run_id요청 단위 추적
status=FAILED상태 조회에서 실패 확인
failed_node어느 단계에서 실패했는지 확인
error_typetimeout, connection, validation 분류
error_message원인 분석의 시작점
partial_result실패 전까지의 분석 결과 보존
OS/CPU Agent와 Lock Agent가 성공하고 SQL Agent가 실패했다면, 전체를 버리지 말고 어디까지 성공했는지 남겨야 합니다.

핵심 정리

오늘 가져갈 여섯 가지

운영 구조workflow만으로는 서비스가 아닙니다.
비동기 실행API 요청과 Worker 실행을 분리합니다.
ReAct 운영step 추적, 실행 예산, Tool 정책이 필요합니다.
