LLM은 언어와 패턴을 다루는 데 강합니다. 핵심은 자유로운 입력을 의미 있는 구조와 작업 단위로 바꾸는 능력입니다.

LLM이 약한 것

최신 사실학습 시점과 입력 컨텍스트 밖의 최신 상태를 스스로 알 수 없습니다.
시스템 상태관측하지 않은 상태를 추측할 수 있습니다.
환각사실 검증보다 자연스럽고 가능성 높은 문장 생성에 최적화되어 있습니다.
긴 문맥모든 제약을 같은 중요도로 유지하기 어렵습니다.
수치 계산계산기처럼 동작하지 않고 다음 토큰을 확률적으로 생성합니다.
원인 구분가능성 높은 설명을 확정 결론처럼 표현할 수 있습니다.

최신성 / 외부 상태 보완

DB, Metric, Log 조회 Tool을 연결하고, 조회 시점과 시간 범위를 명시하며, Tool 호출 결과를 LLM 입력에 포함합니다.

예: “지금 결제 API 장애인가요?”

위험

  • 일반적인 원인 추측
  • 과거 지식으로 현재 상태 단정

설계 보완

  • Metric Tool: 최근 30분 error rate / latency 조회
  • Log Tool: 최근 배포 이후 timeout, 5xx 로그 조회
  • Deploy DB: 마지막 배포 시각 확인
  • 답변에 조회 시점과 시간 범위 명시

답변 형태

“14:10 기준, 최근 30분 결제 API 5xx 비율이 0.2%에서 4.8%로 증가했습니다.
마지막 배포는 13:42이며, 배포 직후부터 timeout 로그가 증가했습니다.”

환각 보완

근거 중심 Prompt를 작성하고, 원인 후보와 확정 사실을 구분하며, 최종 답변 검증 레이어를 추가합니다.

환각을 줄이는 설계 패턴

근거 제한

  • Tool/RAG 결과 안에서만 사실을 말하도록 지시
  • 근거 문서나 로그가 없으면 “확인 불가”로 표시
  • 답변에 사용한 데이터 소스와 조회 범위 포함

사실 / 후보 분리

  • 확인된 사실: 관측된 Metric, Log, DB 결과
  • 원인 후보: 사실에서 추론한 가능성
  • 추가 확인 필요: 아직 조회하지 못한 항목

검증 레이어

  • 최종 답변 전 근거 없는 단정 표현 검사
  • 수치, 시간, 시스템명을 Tool 결과와 재대조
  • 불확실한 결론은 조건부 표현으로 낮춰 작성

문맥 누락 보완

입력 데이터를 구조화하고, State로 중간 결과를 관리하며, Prompt를 짧고 명확한 역할별 단계로 분리합니다.

긴 문맥을 잃지 않는 설계 패턴

입력 구조화

  • 사용자 요청을 서비스명, 시간 범위, 증상, 제약 조건으로 분해
  • 필수값이 없으면 추측하지 않고 추가 질문 생성
  • 기본값을 사용한 경우 최종 답변에 명시

State 관리

  • Tool 호출 결과와 중간 판단을 Agent State에 저장
  • 확인된 사실, 후보 원인, 미확인 항목을 분리해 유지
  • 다음 단계 Prompt에는 필요한 상태만 요약해 전달

단계 분리

  • 분석, 조회, 판단, 답변 생성을 역할별 Prompt로 분리
  • 각 단계의 출력 Schema를 고정해 누락 필드를 감지
  • 최종 답변 전 요청 조건이 모두 반영됐는지 점검

수치 계산 / 단위 오류 보완

계산은 LLM 전달 전 단계에서 수행하고, rule-based check를 추가하며, 수치 결과와 자연어 해석을 역할로 분리합니다.

Latency 증가율

“결제 API latency가 어제보다 얼마나 늘었나요?”

위험

  • ms와 s 단위 혼동
  • 평균 latency와 p95 latency 혼합
  • 증가율 계산에서 분모/반올림 오류

설계 보완

Metric Tool이 현재값과 기준값을 조회하고, Calculation Layer가 증가율을 계산합니다. LLM은 결과 해석만 담당합니다.

답변

최근 1시간 평균 latency는 240ms, 어제 같은 시간대는 180ms입니다.
증가율은 33.3%이며, p95는 평균과 섞지 않았습니다.

DB 용량 예측

“로그 테이블이 하루 18GB씩 늘면 30일 뒤 얼마나 필요하죠?”

위험

  • GB와 GiB 단위 혼동
  • 보관 기간, 압축률, 인덱스 용량 누락
  • 일 단위 누적 계산에서 기간 포함 오류

설계 보완

Storage Calculator가 증가량, 보관 기간, 인덱스 배율을 계산하고, rule-based check가 단위와 범위를 검증합니다.

답변

데이터만 기준으로 30일 누적 증가량은 540GB입니다.
인덱스 30%를 포함하면 약 702GB를 예약해야 합니다.

모호한 질문 보완

필수 파라미터를 정의하고, 정보 부족 시 추가 질문을 생성하며, 기본값을 사용한 경우 답변에 명시합니다.

예: “어제 밤 결제 API가 느렸던 원인 찾아줘”

필수 파라미터

  • 서비스명, 환경, 시간 범위, 기준 지표를 명시
  • “느림”을 평균, p95, timeout, error rate 중 하나로 변환
  • 분석 범위를 API, DB, 배포, 트래픽 중 어디까지 볼지 결정

추가 질문

  • 필수값이 없으면 한 번에 1-2개만 구체적으로 질문
  • 예: “어제 밤은 22:00-02:00 기준으로 보면 될까요?”
  • 예: “결제 API production 환경을 기준으로 분석할까요?”

기본값 명시

  • 답변 지연이 더 위험하면 안전한 기본값으로 먼저 조회
  • 기본값: production, 최근 장애 기준 시간대, p95 latency
  • 최종 답변에 사용한 기본값과 한계를 함께 표시

도메인 지식 보완

범용 데이터로 학습된 모델은 사내 운영 기준, 서비스별 정책, 장애 대응 절차를 기본적으로 알지 못합니다. Runbook/RAG, 운영 기준, 도메인별 Prompt를 통해 판단에 필요한 지식을 연결합니다.

범용 모델에 도메인 지식을 연결하는 방법

운영 기준 정리

  • 장애, 경고, 정상 상태의 threshold를 명시
  • 서비스별 SLO, error budget, 배포 정책을 정리
  • 운영팀이 쓰는 용어와 판단 기준을 문서화

Runbook / RAG 연결

  • 서비스별 장애 대응 Runbook을 검색 가능한 문서로 연결
  • 운영 정책, 예외 조건, 승인 절차를 답변 근거로 사용
  • 문서 버전과 출처를 함께 남겨 기준 변경을 추적

Prompt에 판단 기준 포함

  • Agent별 Prompt에 도메인 기준과 금지된 단정을 포함
  • threshold를 넘은 경우와 애매한 경우의 응답 규칙 분리
  • 기준이 없으면 임의 판단 대신 “기준 미정”으로 표시

structured output과 Schema를 사용하고, Prompt/model version을 기록하며, 평가셋 기반 회귀 테스트를 수행합니다.

같은 입력에 같은 기준으로 답하게 만드는 방법

Schema 고정

  • 답변을 summary, evidence, diagnosis, next_action 필드로 구조화
  • 필수 필드가 비어 있으면 최종 답변 전 실패 처리
  • 확정 사실과 추론 결과를 서로 다른 필드로 분리

Version 기록

  • Prompt version, model version, Tool version을 로그에 저장
  • 품질 변화가 생겼을 때 어떤 변경이 원인인지 추적
  • 배포 전후 답변 차이를 비교할 수 있게 보관

회귀 테스트

  • 자주 나오는 질문과 장애 사례를 평가셋으로 축적
  • 정답 포맷, 필수 근거, 금지 표현을 자동 검사
  • Prompt/model 변경 전후 품질 저하를 배포 전에 감지

실패 패턴을 설계 변경으로 반영

오답을 모아보면 Prompt 문구만의 문제가 아니라, 어떤 Tool을 언제 조회하고 어떤 형식으로 판단할지의 문제로 드러납니다.

판단 기준 추가상관관계만으로 원인을 확정하지 않도록 규칙화
답변 톤 조정근거가 부족하면 단정 대신 추가 확인 항목을 제시

DB pool 고갈 장애를 Runbook/RAG 지식으로 만드는 3단계

1. 사후 원인 확인

  • 결제 API 5xx와 latency 증가
  • DB connection pool saturation 확인
  • 배포는 같은 시간대였지만 직접 원인은 아님

2. Runbook 규칙 추가

  • 배포 단정 전 DB pool usage와 connection timeout 확인
  • rollback 판단 전 DB access pattern 변경 여부 확인
  • owner, updated_at, 문서 version 기록

3. RAG 검색 문서화

  • service: payment-api
  • symptom: 5xx 증가, latency 증가
  • metric: db.pool.usage, connection_timeout
  • action: pool 설정 확인, DB 부하 확인, scale-out

  • 배포 시점만으로 rollback을 확정하지 않음

ReAct 구현 2: StateGraph 직접 구성

실행 흐름을 직접 제어하고 싶을 때는 State, Node, Conditional Edge를 명시합니다.

from typing import TypedDict
from langgraph.graph import StateGraph, START, END

class AgentState(TypedDict):
    messages: list

def call_llm(state):
    ai = llm.invoke(state["messages"])
    return {"messages": state["messages"] + [ai]}

def call_tools(state):
    ai = state["messages"][-1]
    return {"messages": state["messages"] + run_tools(ai.tool_calls)}

def next_step(state):
    ai = state["messages"][-1]
    return "tools" if ai.tool_calls else END
graph = StateGraph(AgentState)
graph.add_node("llm", call_llm)
graph.add_node("tools", call_tools)

graph.add_edge(START, "llm")
graph.add_conditional_edges(
    "llm",
    next_step,
    {"tools": "tools", END: END},
)
graph.add_edge("tools", "llm")

agent = graph.compile()

장점: 검증, 승인, 재시도, 로그 수집 같은 운영 제어 지점을 graph에 직접 끼워 넣을 수 있습니다.

MCP = Model Context Protocol. 핵심은 Tool을 애플리케이션 내부 함수가 아니라 표준 인터페이스를 가진 외부 능력으로 분리하는 것입니다.

  • Prompt: 재사용 가능한 지시 템플릿

├─ Tool
├─ Resource
└─ Prompt

Client

모델 중심 사고 vs Harness 중심 사고

Model-first

좋은 모델을 고르면 품질이 좋아진다

프롬프트를 잘 쓰면 해결되고, 답변이 틀리면 모델을 바꾸는 방식으로 접근합니다.

Harness-first

모델이 일할 환경을 설계한다

모델이 볼 데이터와 Tool, 실패·검증·승인 흐름, 평가셋과 Runbook/Skill 축적 구조를 함께 설계합니다.

왜 Harness Engineering이 중요해졌는가

모델 성능 상향 평준화좋은 모델을 쓰는 것만으로는 차별화가 어려워졌습니다.
Agentic application 확산모델이 답변을 넘어 Tool을 사용하고 실제 작업을 수행합니다.
운영 신뢰성 요구 증가실패 처리, 권한, 승인, 추적, 평가가 제품 품질을 좌우합니다.
개선 루프의 중요성실패 사례를 평가셋, Runbook, Skill로 축적해야 품질이 유지됩니다.

Harness Engineering

Harness Engineering은 모델 주변 실행 환경을 설계하는 관점입니다.
품질은 모델 밖의 데이터 연결, Tool 설계, Prompt/Schema, Logging, Evaluation, 권한 관리, 검증 루프에서 결정됩니다.
