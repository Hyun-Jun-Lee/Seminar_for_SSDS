Model fit

LLM이 잘하는 것

LLM은 언어와 패턴을 다루는 데 강합니다. 핵심은 자유로운 입력을 의미 있는 구조와 작업 단위로 바꾸는 능력입니다.

의도 파악자연어 질문에서 사용자의 목적을 해석
구조화비정형 입력을 필드, 분류, JSON 형태로 변환
요약긴 문맥에서 핵심을 압축
분류·설명텍스트의 유형을 나누고 이유를 설명
코드 보조생성, 변환, 패턴 해석을 지원
텍스트 변환형식, 톤, 언어, 구조를 목적에 맞게 변경

Risk surface

LLM이 약한 것

최신 사실학습 시점과 입력 컨텍스트 밖의 최신 상태를 스스로 알 수 없습니다.
시스템 상태관측하지 않은 상태를 추측할 수 있습니다.
환각사실 검증보다 자연스럽고 가능성 높은 문장 생성에 최적화되어 있습니다.
긴 문맥모든 제약을 같은 중요도로 유지하기 어렵습니다.
수치 계산계산기처럼 동작하지 않고 다음 토큰을 확률적으로 생성합니다.
원인 구분가능성 높은 설명을 확정 결론처럼 표현할 수 있습니다.

System compensation

최신성·환각·문맥 누락 보완

1. 최신성 / 외부 상태

DB, Metric, Log 조회 Tool을 연결하고, 조회 시점과 시간 범위를 명시하며, Tool 호출 결과를 LLM 입력에 포함합니다.

2. 환각

근거 중심 Prompt를 작성하고, 원인 후보와 확정 사실을 구분하며, 최종 답변 검증 레이어를 추가합니다.

3. 문맥 누락

입력 데이터를 구조화하고, State로 중간 결과를 관리하며, Prompt를 짧고 명확한 역할별 단계로 분리합니다.

Guardrails

계산 오류와 모호한 질문 보완

4. 수치 계산 / 단위 오류

계산과 해석을 분리합니다.

계산은 LLM 전달 전 단계에서 수행하고, rule-based check를 추가하며, 수치 결과와 자연어 해석을 역할로 분리합니다.

5. 모호한 질문

부족한 조건을 명시적으로 다룹니다.

필수 파라미터를 정의하고, 정보 부족 시 추가 질문을 생성하며, 기본값을 사용한 경우 답변에 명시합니다.

운영 품질의 기준

도메인 기준과 답변 일관성 보완

사내 기준, Runbook/RAG, structured output, Schema, Prompt/model version, 평가셋 기반 회귀 테스트가 답변 품질을 반복 가능하게 만듭니다.

Service design

AI 서비스 설계란 무엇인가

AI 서비스 개발에서 중요한 설계는 단순히 모델을 고르는 일이 아닙니다. LLM을 소프트웨어 시스템 안에 어떻게 배치할지 결정하는 일입니다.

데이터어떤 데이터를 LLM에 줄 것인가
Tool어떤 Tool을 붙일 것인가
StateState를 어떻게 관리할 것인가
Failure loop실패, 재시도, 검증을 어떻게 설계할 것인가

Design checklist

AI 서비스 설계의 추가 질문

Prompt / Schema어떻게 구조화할 것인가
Log / Evaluation어떻게 남기고 평가할 것인가
Permission / Approval권한과 승인 흐름을 어떻게 둘 것인가
Feedback loop사용자의 피드백을 어떻게 다시 개선에 사용할 것인가

따라서 AI 서비스 개발에는 프롬프트 작성 능력뿐 아니라 소프트웨어 설계와 구현 역량이 함께 필요합니다.

Skill mindset

반복 가능한 작업 방식도 설계해야 한다

Tool, State, Logging만으로 Agent 품질이 자동으로 안정화되지는 않습니다. 운영에서는 Agent가 매번 같은 기준과 절차로 일하도록 작업 방식을 명시해야 합니다.

분석 순서어떤 순서로 문제를 분석할 것인가
리뷰 기준어떤 기준으로 코드를 리뷰할 것인가
완료 기준어떤 테스트를 통과해야 완료로 볼 것인가
승인 조건어떤 상황에서 사용자 승인이나 추가 질문이 필요한가

Grounding

RAG 활용 방안

RAG는 LLM이 모든 지식을 내부에 기억하도록 만드는 방식이 아닙니다. 필요한 시점에 외부 지식과 근거 문서를 검색해 답변에 반영하는 구조입니다.

운영 근거 검색장애 대응 Runbook, 사내 DB 운영 매뉴얼, 정기 점검 절차
사례 기반 응답과거 장애 사례, 장애 보고서 템플릿, FAQ와 문의 이력
기술 지식 보강SQL 튜닝 가이드, 서비스별 운영 정책
줄이는 문제최신성 한계, 도메인 기준 부족, 근거 부족

Protocol boundary

MCP 개념

MCP = Model Context Protocol. 핵심은 Tool을 애플리케이션 내부 함수가 아니라 표준 인터페이스를 가진 외부 능력으로 분리하는 것입니다.

MCP Server가 제공하는 것

  • Tool: 모델이 호출할 수 있는 기능
  • Resource: 모델이 참고할 수 있는 데이터
  • Prompt: 재사용 가능한 지시 템플릿

MCP 연결 구조

Server
├─ Tool
├─ Resource
└─ Prompt

Client
└─ Server에 연결해
   기능을 모델/Agent에게 노출

When to split agents

A2A는 언제 필요한가

독립 배포Agent를 팀/도메인별로 독립 배포해야 할 때
명확한 책임 경계DB Agent, OS Agent, Log Agent처럼 역할이 분리될 때
이기종 연결서로 다른 프레임워크나 시스템의 Agent를 연결해야 할 때
전문 Agent 위임중앙 Orchestrator가 전문 Agent에게 작업을 위임해야 할 때
운영 표준Agent 간 인증, 메시지 표준, 장애 전파 관리가 필요할 때
Closing frame

Harness Engineering

Harness Engineering은 모델 주변 실행 환경을 설계하는 관점입니다.
품질은 모델 밖의 데이터 연결, Tool 설계, Prompt/Schema, Logging, Evaluation, 권한 관리, 검증 루프에서 결정됩니다.

01

모델 상향 평준화

좋은 모델만으로는 차별화가 어렵습니다.

02

병목은 모델 밖

데이터, Tool, 평가, 권한이 안정성을 결정합니다.

03

Agentic application

State, Workflow, Verification, Observability가 필수입니다.

04

개선 루프

실패 사례를 축적하고 다시 개선하는 구조가 필요합니다.
