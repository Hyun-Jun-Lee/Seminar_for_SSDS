Background

인증은 “비밀번호 확인”보다 더 넓은 문제예요

앱에서 인증은 보통 로그인 화면 하나처럼 보입니다. 하지만 시스템 안쪽에서는 훨씬 많은 일이 일어나요. 사용자가 누구인지 증명하고, 토큰이 아직 유효한지 확인하고, 만료되면 갱신하고, 서버가 사용자별 데이터를 안전하게 분리해야 합니다. Curitalk에서는 특히 대화 기록, 검색 준비 카드, 문법 피드백이 모두 사용자별로 묶이기 때문에 인증은 제품 데이터의 울타리 역할을 해요.

기존 구조에서는 FastAPI가 이 울타리의 대부분을 직접 들고 있었습니다. 이메일/비밀번호 로그인, Google 모바일 로그인, 개발 토큰, refresh token rotation, logout revoke까지 백엔드가 직접 처리했고, Flutter 앱은 그 결과로 받은 자체 access_token과 refresh_token을 저장했습니다.

FlutterGoogle SDK에서 id token 획득
FastAPI/auth/google/mobile로 Google token 검증
DB users앱 사용자 row 생성 또는 조회
FastAPI JWT자체 access/refresh token pair 발급
API 요청FastAPI JWT를 Bearer로 전달
기존 구조의 부담 로그인 제공자가 늘어날수록 FastAPI가 OAuth 세부사항, token refresh, revoke, 보안 저장 정책을 계속 떠안게 됩니다. 모바일 앱 배포 전에는 가능하지만, 실제 운영으로 갈수록 “영어 회화 제품”보다 “인증 서버”를 만드는 비중이 커져요.

이번 변경의 좁은 배경: Supabase를 세션 권한자로 삼기

Supabase Auth를 쓰면 Google 로그인과 세션 갱신은 Supabase SDK가 맡고, FastAPI는 Supabase가 발급한 access token만 검증하면 됩니다. 말하자면 여권 발급소는 Supabase이고, FastAPI는 입구에서 여권을 확인하는 서비스 게이트가 됩니다. FastAPI는 여전히 중요한 역할을 해요. LLM API 키, 검색 품질 판단, 대화 생성, 문법 피드백 같은 서버 전용 로직은 계속 FastAPI 뒤에 남아 있습니다.

영역 이전 이후
세션 권한자 FastAPI 자체 JWT와 refresh token Supabase Auth session
모바일 로그인 Google id token을 FastAPI에 보내 자체 token pair 발급 Google id/access token으로 Supabase session 생성
사용자 테이블 users와 refresh_tokens profiles, id는 Supabase user id
백엔드 인증 API register, login, dev token, google/mobile, refresh, logout, me primary path는 GET /api/auth/me
Intuition

핵심 직관은 “토큰을 만들지 말고, 신뢰할 토큰을 확인하자”예요

이 변경의 본질은 책임의 재배치입니다. FastAPI는 더 이상 사용자의 로그인 세션을 직접 제조하지 않습니다. 대신 Supabase가 만든 access token을 받아서 “이 토큰이 우리 Supabase 프로젝트의 유효한 사용자에게 속하는가?”를 확인합니다. 확인이 끝나면 그 사용자의 앱 프로필을 profiles에 맞춰 생성하거나 갱신합니다.

사용자 경험은 더 단순해져요

사용자는 여전히 Google 버튼 하나만 누릅니다. 하지만 앱 내부에서는 자체 refresh token 저장소가 아니라 Supabase SDK의 session이 source of truth가 됩니다. 그래서 앱 재시작, token refresh, sign-out의 의미가 모바일 SDK와 더 자연스럽게 맞물립니다.

중요한 엣지 케이스 이 구현은 FastAPI에서 Supabase /auth/v1/user를 호출하는 원격 검증 방식이에요. Supabase 프로젝트가 어떤 JWT signing mode를 쓰든 비교적 안전하게 동작하지만, 보호 API 호출마다 네트워크 확인 비용이 생깁니다. 운영 트래픽이 커지면 JWKS 기반 로컬 검증으로 최적화할 수 있어요.

1. 설정: Supabase project를 명시적으로 요구하기

backend/config.py에는 Supabase URL, publishable key, 원격 검증 timeout, 그리고 AUTO_CREATE_TABLES가 추가됐습니다. 특히 required_supabase_publishable_key와 normalized_supabase_url은 누락된 환경변수를 조용히 넘기지 않고 빠르게 실패하게 만듭니다. 

supabase_url: str | None = None
supabase_publishable_key: str | None = None
supabase_auth_verify_mode: Literal["remote"] = "remote"
supabase_auth_timeout_seconds: float = 5.0
auto_create_tables: bool = False

backend/main.py에서는 Base.metadata.create_all을 기본 실행하지 않도록 했어요. 이제 스키마는 Alembic migration으로 관리한다는 뜻입니다.

2. 백엔드 Auth: 자체 token 발급기를 걷어내고 verifier로 축소

가장 큰 삭제는 backend/domains/auth/service.py와 router.py에서 일어났습니다. 비밀번호 hash, JWT encode/decode, refresh token hash, Google callback, dev token 발급 로직이 primary path에서 제거됐어요. 대신 서비스는 Supabase claim을 앱 프로필로 연결하는 얇은 계층이 되었습니다. 

@dataclass(frozen=True)
class SupabaseUserClaims:
    sub: str
    email: str
    name: str
    oauth_provider: str | None = None
    avatar_url: str | None = None

실제 token 검증은 새 파일 backend/domains/auth/supabase.py에 있습니다. FastAPI가 받은 Bearer token을 Supabase Auth 서버의 /user endpoint로 확인하고, 응답에서 id, email, name, provider, avatar를 뽑아 앱 claim으로 바꿉니다. 

response = await client.get(
    f"{settings.supabase_auth_url}/user",
    headers={
        "apikey": settings.required_supabase_publishable_key,
        "Authorization": f"Bearer {token}",
    },
)
왜 router가 작아졌을까요? 모바일의 primary 인증 흐름이 Supabase로 이동했기 때문에 FastAPI의 Auth router는 이제 GET /api/auth/me에 집중합니다. “로그인”이 아니라 “현재 Supabase 사용자에 대응하는 앱 프로필을 반환”하는 API가 된 거예요.

3. DB: users가 아니라 profiles를 앱 소유 데이터로 삼기

Supabase에는 내부 auth.users가 있습니다. 앱이 직접 그 테이블을 소유하려고 하면 경계가 흐려집니다. 그래서 앱의 표시 이름, 활성 상태, avatar 같은 확장 정보는 profiles에 둡니다. profiles.id는 Supabase user id와 같고, conversation의 user_id도 profiles를 참조합니다. 

table profiles {
  id: UUID PRIMARY KEY        // Supabase auth.users.id와 같은 값
  email: STRING UNIQUE
  name: STRING
  is_active: BOOLEAN
  oauth_provider: STRING?
  avatar_url: STRING?
}

migration에는 안전장치가 들어 있습니다. 기존 users나 conversations에 데이터가 있으면 migration이 실패합니다. 데이터가 있는 운영 DB에서는 그냥 drop하면 안 되고, 별도 backfill migration이 필요하기 때문입니다. 

if user_count or conversation_count:
    raise RuntimeError(
        "profiles migration requires an empty pre-Supabase users/conversations dataset."
    )

4. Flutter: token storage가 아니라 Supabase session을 읽기

Flutter에서는 supabase_flutter가 추가되고, 앱 시작 시 Supabase.initialize가 실행됩니다. AppConfig는 SUPABASE_URL과 SUPABASE_PUBLISHABLE_KEY를 필수 dart-define 값으로 요구합니다. 

await Supabase.initialize(
  url: AppConfig.requiredSupabaseUrl,
  publishableKey: AppConfig.requiredSupabasePublishableKey,
);

새 SupabaseAuthService는 세 가지 역할을 겸합니다. 현재 access token을 읽고, Supabase session을 refresh하고, Google token으로 sign-in합니다. 그래서 Dio interceptor는 더 이상 secure storage에서 FastAPI token pair를 읽지 않습니다. 

abstract interface class SupabaseAuthService
    implements AccessTokenProvider, SessionRefreshProvider {
  Stream<SupabaseSessionChange> get authStateChanges;
  Future<bool> hasCurrentSession();
  Future<void> signInWithGoogleTokens({
    required String idToken,
    required String accessToken,
  });
  Future<void> signOut();
}

5. 401 처리: /auth/refresh 대신 Supabase refreshSession

이전에는 API가 401을 반환하면 FastAPI의 /auth/refresh를 호출해 token pair를 rotate했습니다. 이제는 Supabase SDK의 refreshSession()을 호출하고, 새 access token으로 원 요청을 한 번 재시도합니다. 동시 401이 여러 개 와도 하나의 refresh 작업을 공유하는 구조는 유지했습니다.

컴포넌트 변경 요약
AuthTokenInterceptor TokenStorage 대신 AccessTokenProvider에서 Supabase access token을 읽음
TokenRefreshInterceptor /auth/refresh POST 제거, SessionRefreshProvider.refreshAccessToken 사용
AuthController 앱 시작 시 Supabase current session 확인 후 /auth/me 조회
GoogleIdentityService Google idToken뿐 아니라 Supabase sign-in에 필요한 accessToken도 반환

6. 문서와 테스트: 새 계약을 한 번에 맞추기

README.md, docs/DSL.md, mobile/README.md, .agent/architecture.md가 Supabase Auth 기준으로 갱신됐습니다. 테스트도 old local JWT/dev-token 중심에서 Supabase verifier, profiles repository, Supabase session 기반 Flutter controller 테스트로 바뀌었습니다.

Quiz

이 변경을 제대로 이해했는지 확인해볼까요?

각 문제는 한 가지 답만 맞습니다. 버튼을 누르면 바로 피드백이 나와요.

1. 이번 변경 후 모바일 앱의 primary 세션 source of truth는 무엇인가요?

2. FastAPI가 보호 API 요청에서 Supabase access token을 검증하는 현재 방식은 무엇인가요?

3. profiles 테이블을 도입한 이유로 가장 알맞은 것은 무엇인가요?

4. API가 401을 반환했을 때 Flutter의 새 refresh 흐름은 어떻게 동작하나요?

5. migration이 기존 users/conversations 데이터가 있으면 실패하도록 한 이유는 무엇인가요?

한 줄로 요약하면, 이번 diff는 Curitalk이 직접 인증 서버가 되려던 길에서 한 발 물러나 Supabase Auth를 신뢰 경계의 앞단에 세운 변경입니다. 그 덕분에 FastAPI는 제품의 고유한 일, 즉 관심사 기반 영어 대화와 AI 피드백에 더 집중할 수 있게 됐어요.
