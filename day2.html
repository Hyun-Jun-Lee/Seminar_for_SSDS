Day 2

DaOps로 보는
LangGraph Agent 설계

운영 업무를 Multi-Agent Workflow로 바꾸는 방법

오늘의 질문

“DB가 느립니다”를 Agent는 어떻게 처리해야 할까요?

사용자 요청

DB가 느립니다. 원인을 분석해 주세요.

Agent가 해야 할 일

질문을 분류하고, 필요한 지표를 조회하고, 여러 분석 결과를 합쳐 근거 중심 진단을 만들어야 합니다.

2일차는 Agent를 “답변 생성기”가 아니라 운영 분석 workflow로 보는 시간입니다.

DaOps 문제 정의

질문이 단순해도 확인해야 할 원인은 여러 도메인에 걸쳐 있다.

DB 내부Wait Event, DB Time, Lock, SQL 실행 시간
Host 상태CPU Util, Load Average, Run Queue, Memory Pressure
용량과 리소스TEMP, UNDO, PGA, Session, Tablespace
시간 범위지금인지, 최근 10분인지, 배포 이후인지
조치 위험Kill Session, Resource 증설, SQL 튜닝의 영향
불확실성확정 원인인지, 후보인지, 추가 확인이 필요한지

Multi-Agent 관점

중요한 것은 Agent 개수가 아니라 책임 분리와 호출 기준입니다.

1. 분류

요청 유형과 필요한 분석 범위를 정합니다.

2. 선분석

전체 상태를 보고 이상 후보를 좁힙니다.

3. 전문 분석

필요한 Domain Agent만 실행합니다.

4. 병합

분석 결과가 모두 도착했는지 확인합니다.

5. 검증

근거, 누락, 위험 조치를 점검합니다.

설계 질문: “어떤 Agent를 만들까?”보다 “언제 어떤 Agent를 호출하고 결과를 어떻게 합칠까?”가 더 중요합니다.

Multi-Agent 관점

Agent 분리 기준

도메인 책임CPU, SQL, Lock, TEMP처럼 해석 기준과 조회 지표가 다른가?
Tool 경계사용하는 조회 함수, 권한, 데이터 소스가 다른가?
판단 기준원인 판단 규칙이나 위험도 기준이 다른가?
실행 비용항상 실행하면 비싸서 필요할 때만 호출해야 하는가?
출력 SchemaSummary가 합칠 수 있는 독립적인 reason_data를 만들 수 있는가?
운영 책임장애 대응이나 조치 승인 주체가 다른가?
Agent는 기능 단위로 많이 쪼개는 것이 아니라, 책임·Tool·판단 기준·출력 형태가 달라지는 지점에서 분리합니다.

DaOps 전체 흐름

분류 → 전체 상태 확인 → 세부 분석 → 결과 병합 → 진단 요약 → 품질 검증

START
  ↓
사용자 질문 / 요청 파라미터 수집
  ↓
Classifier
  ├─ 특정 도메인 질문 → Domain Sub Agent 직접 호출
  └─ 넓은 성능 저하 질문 → Global Health
                            ↓
                    필요한 Domain Sub Agent 선택
                            ↓
                    Fan-out: 여러 Agent 병렬 실행
                            ↓
                    Join: 결과 병합 / 완료 여부 확인
                            ↓
                    Diagnosis Summary
                            ↓
                    Quality Gate
                            ↓
                    Final Response

Part 1

LangGraph로
Workflow 구조화하기

LangGraph

조건부 실행, 상태 공유, 병렬 분석이 필요해지면 그래프 구조가 유리합니다.

상태 공유여러 Node가 user_question, target_nodes, reason_data를 함께 사용합니다.
조건부 라우팅request_type과 분석 결과에 따라 다음 Node가 달라집니다.
병렬 실행Global Health 이후 필요한 Domain Agent를 fan-out 할 수 있습니다.
안전한 병합모든 결과가 도착했는지 확인한 뒤 Summary를 실행합니다.
중복 방지visited_nodes와 flag로 같은 Node의 반복 실행을 막습니다.
추적 가능성각 Node 실행과 상태 변화를 로그로 남기기 좋습니다.

LangGraph

LangGraph는 State, Node, Edge, Conditional Edge로 workflow를 표현합니다.

구성요소역할DaOps 예시
Stateworkflow 전체가 공유하는 실행 문맥user_question, target_nodes, reason_data
Node하나의 처리 단계Classifier, Global Health, Temp Agent
Edge다음 단계로 이동하는 연결Classifier → Global Health
Conditional Edge상태를 보고 다음 경로 선택request_type에 따라 Agent 선택

LangGraph

가장 단순한 형태는 Node를 등록하고, Edge와 조건부 Edge를 연결하는 구조입니다.

from langgraph.graph import StateGraph, START, END

graph = StateGraph(dict)
graph.add_node("classifier", classifier_node)
graph.add_node("global_health", global_health_node)
graph.add_node("summary", diagnosis_summary_node)

graph.add_edge(START, "classifier")
graph.add_conditional_edges("classifier", route_after_classifier)
graph.add_conditional_edges("global_health", route_to_domain_agents)
graph.add_edge("summary", END)

app = graph.compile()

State 설계

MainState는 workflow의 기억 장치이자 제어판입니다.

필드의미제어에 쓰이는 방식
user_question사용자의 원본 질문분류와 최종 답변의 기준
request_type질문 분류 결과Global Health 여부 결정
target_nodes실행해야 할 Agent 목록Join 완료 조건
visited_nodes이미 완료된 Agent 목록중복 실행 방지
reason_data각 Agent가 만든 근거 데이터Summary 입력
diagnosis_summary_started요약 실행 여부Summary 중복 실행 방지

State 설계

class MainState(TypedDict):
    user_question: str
    request_type: str
    target_nodes: list[str]
    global_health_target_nodes: list[str]
    visited_nodes: list[str]
    reason_data: list[dict]
    diagnosis_summary: dict | None
    diagnosis_summary_started: bool
    reason_join_ready: bool
    quality_gate_result: dict | None
    final_answer: str | None
Multi-Agent workflow에서는 “무엇을 실행했는지”, “무엇이 끝났는지”, “무엇을 합쳐야 하는지”를 State로 관리해야 합니다.

Part 2

DaOps Node
설계하기

Classifier Node

Classifier는 답변하지 않고, 질문의 성격과 다음 실행 경로를 정합니다.

분류 예시

  • “TEMP가 부족한가요?” → Temp Agent 직접 호출
  • “DB가 느립니다” → Global Health 먼저 실행
  • “최근 10분 CPU 병목인가요?” → OS/CPU Agent 우선 호출

결정해야 할 것

  • request_type은 무엇인가?
  • 전체 상태 점검이 필요한가?
  • 바로 호출할 Domain Agent가 있는가?
  • 분류 결과를 신뢰할 수 있는가?

Global Health Agent

Global Health는 분석 범위를 좁히는 Agent입니다.

확인 항목

  • Host CPU Util %, Load Average
  • DB Wait Time Ratio, DB CPU Time Ratio
  • DB Time/sec, AAS
  • Top Event, 주요 Metric 변화

결과 예시

Global Health 결과
  - Host CPU Util: 92%
  - DB CPU Time Ratio: 높음
  - Top Wait Event: CPU + db file sequential read

선택할 Sub Agent
  - OS/CPU Agent
  - SQL Agent

Domain Sub Agent

좋은 Domain Agent는 책임 경계가 명확합니다.

책임 경계가 명확해야 Tool 호출, 근거 생성, 판단 범위를 통제할 수 있습니다. 경계가 흐리면 중복 조회, 역할 충돌, Summary 단계의 판단 불일치가 발생합니다.
Temp AgentTEMP 사용량, spill, time_to_full 분석
Undo Agentlong transaction, undo retention, rollback 이슈 분석
Lock Agentblocking session, wait chain, TX/TM lock 분석
OS/CPU AgentHost CPU, DB CPU, run queue, load average 분석
SQL AgentTop SQL, 실행 계획, elapsed time 변화 분석
Memory/PGA AgentPGA 사용량, memory pressure, process별 사용량 분석

Part 3

Sub Agent를
ReAct로 전환하기

Sub Agent 전환

기존 구조는 정해진 조회 함수를 모두 호출한 뒤 LLM에게 분석을 맡기는 방식입니다.

고정 조회 흐름

DB가 느립니다
  ↓
CPU 조회
Wait Event 조회
Lock 조회
Top SQL 조회
TEMP 조회
UNDO 조회
  ↓
모든 결과를 LLM에 넣고 분석

생기는 한계

  • 질문과 무관한 데이터까지 항상 조회합니다.
  • Tool 호출 시간과 비용이 고정적으로 커집니다.
  • LLM 입력에 불필요한 문맥이 섞입니다.
  • 중간 결과를 보고 추가 조회하기 어렵습니다.

Sub Agent 전환

ReAct 구조는 관찰 결과를 보면서 다음에 필요한 Tool을 선택합니다.

1. 질문

요청과 시간 범위, 대상 DB를 확인합니다.

2. 가설

현재 가능한 원인 후보를 세웁니다.

3. Tool 선택

필요한 조회 함수만 호출합니다.

4. 관찰

조회 결과로 다음 행동을 결정합니다.

5. 분석

충분한 근거가 모이면 결과를 반환합니다.

LangGraph는 전체 workflow를 제어하고, ReAct는 각 Domain Sub Agent 내부에서 Tool 호출 순서를 결정하게 해줍니다.

Sub Agent 전환

고정 조회 방식과 ReAct 방식은 데이터 수집 전략이 다릅니다.

항목고정 조회 방식ReAct 방식
조회 방식정해진 함수를 모두 호출필요한 Tool을 선택적으로 호출
문맥 구성불필요한 데이터가 섞일 수 있음관찰 결과 중심으로 누적
비용/시간질문과 무관하게 비슷하게 큼질문에 따라 줄어들 수 있음
분석 흐름조회 후 한 번에 분석조회와 판단을 반복
주의점과조회, 문맥 오염반복 제한, Tool 설계, 검증 필요

Sub Agent 전환

예를 들어 SQL Agent는 필요한 근거를 단계적으로 수집하는 ReAct Agent가 될 수 있습니다.

SQL ReAct Agent 예시

1. 최근 10분 Top Wait Event 조회
2. CPU성 대기와 I/O 대기가 함께 높음을 관찰
3. Top SQL 조회
4. 특정 SQL의 elapsed time 증가 확인
5. 실행 계획 변화 여부 조회
6. SQL 병목 후보와 근거 반환

전환 기준

  • Tool은 작고 명확한 책임을 가져야 합니다.
  • Tool 이름과 설명은 LLM이 선택하기 쉬워야 합니다.
  • 시간 범위, 대상 DB, 단위는 명시적으로 전달해야 합니다.
  • 최대 반복 횟수와 종료 조건을 둬야 합니다.
  • 최종 출력은 기존 reason_data schema에 맞춰야 합니다.

ReAct 설계

Tool 설명과 결과값 예시

Tool 설명

LLM이 “어떤 Tool을 언제 호출할지” 고르는 기준입니다. Tool 이름, 호출 상황, 입력값, 반환 요약이 명확해야 합니다.

@tool
def get_top_wait_events(db_id, start_time, end_time):
    """
    DB 지연 원인이 CPU, I/O, Lock, Commit,
    Network 중 어디에 가까운지 분류할 때 사용한다.

    Returns:
      event_name: 대기 이벤트명
      wait_class: User I/O, CPU, Concurrency 등
      db_time_ratio: 전체 DB Time 중 비율, 0~1
      avg_wait_ms: 평균 대기 시간, milliseconds
    """

Tool 결과 Schema

Tool이 반환한 조회 데이터의 의미를 설명하는 schema입니다. 숫자는 의미, 단위, 범위를 함께 전달해야 합니다.

{
  "event_name": "db file sequential read",
  "wait_class": "User I/O",
  "db_time_ratio": 0.42,
  "avg_wait_ms": 18.5,
  "unit": {
    "db_time_ratio": "0~1 ratio",
    "avg_wait_ms": "milliseconds"
  }
}

ReAct 설계

Prompt와 Runbook/RAG 예시

Runbook 검색 예시 질문: TEMP가 계속 증가하는데 위험한가요?
  1. TEMP 사용량 조회
  2. time_to_full이 25분으로 계산됨
  3. search_runbook("TEMP time_to_full 위험 기준") 호출
  4. Runbook에서 “30분 미만이면 즉시 확인 필요” 기준 확인
  5. TEMP 부족 위험도를 high로 판단
  6. 근거와 조치 전 확인 절차를 포함해 답변
Agent Prompt 예시 Top SQL만으로 원인을 확정하지 말고 Wait Event, CPU, I/O 지표와 함께 판단합니다. DB Time 30% 이상이면 주요 후보로 보고, 데이터가 부족하면 “가능성”으로 표현합니다.
Runbook / RAG 예시 CPU 병목 판단 기준, Lock wait 대응 절차, TEMP 부족 시 time_to_full 기준, 위험 조치 전 확인 절차처럼 길고 자주 바뀌는 운영 지식을 둡니다.
Runbook/RAG는 Agent에게 벡터 검색 가능한 Tool로 제공하고, Agent는 지표 조회 결과에 맞는 운영 기준을 필요할 때 검색합니다.

결과 병합과
최종 검증하기

Fan-out / Fan-in

Fan-out

Global Health
  ├─ OS/CPU Agent
  ├─ Lock Agent
  └─ SQL Agent

Fan-in

OS/CPU Agent \
Lock Agent    → Join → Diagnosis Summary
SQL Agent    /
Join은 결과 병합 지점이면서, 누락·중복·조기 실행을 막는 제어 지점입니다.
visited_nodes = ["os_cpu", "lock"]
target_nodes = ["os_cpu", "lock", "sql"]

reason_join_ready = set(visited_nodes) >= set(target_nodes)

if not reason_join_ready:
    return "wait_for_more_results"

if state["diagnosis_summary_started"]:
    return END

return "diagnosis_summary"

Diagnosis Summary 주의사항

Summary Node에서 특히 주의해야 할 점

1. 근거와 결론 분리 각 Agent가 제공한 수치, 시계열, Wait Event, SQL 근거를 먼저 정리하고 그 위에 진단 결론을 얹습니다.
2. Agent 간 충돌 처리 SQL Agent와 OS/CPU Agent가 서로 다른 원인을 지지하면 억지로 하나를 고르지 않고, 공통 근거와 충돌 지점을 분리합니다.
3. 없는 근거 생성 금지 조회하지 않은 지표, 존재하지 않는 SQL ID, 확인하지 않은 장애 원인을 Summary 단계에서 새로 만들지 않습니다.
Summary Node의 역할은 “그럴듯한 문장 생성”이 아니라, Sub Agent들이 만든 근거를 잃지 않고 일관된 진단으로 정리하는 것입니다.

Quality Gate

Quality Gate는 최종 답변의 실패 유형을 처리합니다.

검증 항목확인 질문실패 시 처리
질문 적합성사용자 질문에 직접 답했는가?질문 범위에 맞게 답변을 재작성
근거 반영Sub Agent 결과가 답변에 반영됐는가?누락된 reason_data를 Summary에 다시 반영
근거 일관성Agent 결과와 최종 답변이 모순되지 않는가?충돌 지점을 표시하고 결론을 완화
조치 안전성위험한 조치를 단정적으로 권하지 않는가?조치 전 확인 조건과 승인 절차 추가
불확실성 표현데이터 부족을 명확히 표현했는가?확정 표현을 후보/가능성 표현으로 변경

Quality Gate

Quality Gate는 점수와 필수 통과 조건으로 final_response를 제어할 수 있습니다.

점수화 예시

항목점수의미
질문 적합성0~2질문에 직접 답했는가
근거 반영0~2reason_data가 반영됐는가
근거 일관성0~2Agent 결과와 모순이 없는가
조치 안전성0~2위험 조치를 단정하지 않았는가
불확실성 표현0~2데이터 부족을 표현했는가

통과 조건 예시

  • 총점 8점 이상일 때만 final_response로 전달
  • 조치 안전성은 반드시 2점이어야 통과
  • 근거 일관성은 최소 1점 이상이어야 통과
